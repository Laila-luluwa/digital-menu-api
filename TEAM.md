# Команда и зоны ответственности

## Разработчик A: Архитектура и Back-office
Цель: построить надежный фундамент API, безопасность и административный контур.

1. База данных и изоляция тенантов
- Проектирование и поддержка схемы данных: `restaurants`, `users`, `tables`, `menu_items`, `inventory`, `orders`, `order_items`, `sessions`.
- Tenant isolation: любой запрос к данным ресторана должен фильтроваться по `restaurantId`.
- Поддержка ролевой модели (`OWNER`, `MANAGER`, `KITCHEN`, `PLATFORM_ADMIN`) и связанной авторизации.

2. Admin API
- CRUD для меню, столов и пользователей в защищенном контуре (`/api/*`).
- Контроль остатков (inventory) и обновление данных позиций меню.
- Реализация безопасных операций изменения данных с учетом `restaurantId`.

3. QR и столы
- Логика создания и обслуживания столов (`tableCode`).
- Поддержка сценария старта гостевой сессии через код стола.

## Разработчик B: Product Flow (Diner + Kitchen)
Цель: реализовать полный путь от заказа гостя до обработки на кухне.

1. Diner Flow (гость)
- `POST /api/diner/sessions/start`: запуск гостевой сессии для стола.
- `GET /api/diner/menu`: получение меню в рамках активной сессии.
- `POST /api/diner/orders`: создание заказа с проверкой входных данных и остатков.

2. Kitchen Flow (очередь кухни)
- `GET /api/kitchen/orders`: получение активных заказов.
- `PATCH /api/kitchen/orders/:id/status`: обновление статуса заказа.
- Допустимая цепочка статусов: `QUEUED -> COOKING -> READY -> SERVED`.

3. Real-time
- Оповещения кухни о новых заказах и изменениях статусов через WebSocket (`/api/kitchen/ws`).
- Поддержка стабильной работы real-time канала с проверкой авторизации.

## Общие правила
- Не допускать пересечения данных между ресторанами.
- Все изменения статусов и критичных данных должны быть валидированы.
- Для операций, связанных с остатками и заказами, использовать транзакционный подход.
